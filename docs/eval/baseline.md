# Baseline Snapshot

This document accompanies `tests/fixtures/eval/baseline.json` and explains how the values are produced, what per-category numbers mean, and how to reproduce or verify the snapshot. The methodology is fixed by ADR 0003; the JSON file is the canonical regression-gate input for Phase 3ã€œ6.

## Methodology

The harness composes `rurico` primitives in a `recall`-inspired inline pipeline (`src/eval/pipeline.rs`): an in-memory SQLite store with FTS5 and `sqlite-vec` virtual tables retrieves candidates, RRF merges the two ranked lists with `K = 60`, and the cross-encoder reranker rescores the top-`k` survivors. The fixture under `tests/fixtures/eval/` ships 60 documents (covering seven broad domains: `programming` / `infrastructure` / `ml-ai` / `database` / `web` / `tooling` / `concepts`) and 147 queries grouped into seven IR task categories (`factoid` / `conceptual` / `definitional` / `comparative` / `howto` / `listing` / `troubleshooting`, 21 queries each) annotated with graded relevance scores in `{0, 1, 2, 3}`. Every reported metric (Recall, MRR, nDCG at `k âˆˆ {5, 10}`) carries a 95% bootstrap confidence interval computed over `n = 1000` resamples with `seed = 42` so successive runs produce bit-identical CI bounds, satisfying NFR-002. Three known-answer fixtures (`identity` / `reverse` / `single_doc`) exercised through `tests/eval_smoke.rs` guard the harness wiring against query-document swaps, RRF argument reversal, and `k` off-by-one bugs that a shuffle mutation alone cannot detect.

## Per-category breakdown

`baseline.json::per_category` reports the same metric set, grouped by the seven IR task categories named above. These are emitted for exploratory inspection only â€” Business Rule BR-001 fixes the regression gate on the global metric, never per-category, because per-category sample sizes (21 queries) yield bootstrap half-widths wide enough to mask realistic Phase 3ã€œ6 improvements. Per-category entries whose CI half-width exceeds `0.10` (`UNINFORMATIVE_HALF_WIDTH` in `src/eval/baseline.rs`) carry an `"uninformative": true` flag in the JSON; reviewers should ignore those entries when reasoning about per-task search behaviour and treat their presence as a request to expand the fixture in that specific category before drawing conclusions. The committed JSON also retains the per-category point estimates so trend analysis across Phase 3ã€œ6 stays possible without committing to gate decisions on small samples.

## Reproducibility

Regeneration follows ADR 0002's tolerance contract: `cosine_similarity â‰¥ 0.99999 âˆ§ max_abs_diff â‰¤ 1e-5` between successive runs on the same machine, model revision, and mlx-rs version. The snapshot pins `model_id = "cl-nagoya/ruri-v3-310m"` together with an `mlx_rs_version` field and a fixture-content hash (FNV-1a 64-bit prefixed `fnv1a64:` over the three JSONL files, chosen over SHA-256 to avoid pulling in a cryptographic dependency while still flagging fixture mutation). The file also carries a `schema_version` envelope (currently `"1.0"`) and `kind: "forward"` discriminator so consumers can refuse silently-incompatible files; the sibling reverse-ranker artifact at `tests/fixtures/eval/reverse_baseline.json` carries the same envelope with `kind: "reverse"`. The `captured_with` field records which subcommand produced the file. To recreate the file from a clean clone on Apple Silicon with the embed model cached, run `cargo run --release --features eval-harness --bin eval_harness -- capture-baseline output=tests/fixtures/eval/baseline.json`. To verify that a candidate change leaves the global metrics within tolerance run `eval_harness verify-baseline baseline=tests/fixtures/eval/baseline.json`; exit code zero with the stderr banner `verify-baseline: passed` confirms no drift, exit `1` reports a metric regression naming the metric and observed delta, exit `2` is a usage / validation error, and exit `3` is an infrastructure failure (model load, pipeline crash, baseline I/O). Cross-machine drift outside the tolerance is a Reassessment Trigger documented in ADR 0003 â€” regenerate the baseline in a dedicated PR rather than relax the tolerance.
